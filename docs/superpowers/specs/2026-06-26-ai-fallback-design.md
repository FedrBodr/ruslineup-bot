# Этап 6 — AI-фолбэк (свободный чат с памятью по базе знаний)

_2026-06-26 · FED-243 · ветка `fedrbodr/fed-243-etap-6-ai-folbek-llm-po-baze-znanii`_

## Цель

Бот отвечает на свободные вопросы клиентов через OpenAI по базе знаний бренда, с памятью диалога. Если данных не хватает или вопрос не по теме — мягко эскалирует на менеджера. Расходы под контролем (лимит на пользователя в день), секреты и чувствительные данные — только из ENV.

## Подход

OpenAI **Chat Completions** (официальный `openai` SDK, async-клиент). База знаний небольшая → целиком кладётся в системный промпт, **без эмбеддингов/RAG** (оверкилл для такого объёма). Системный промпт = роль + санитизированный `knowledge.md` + конкретика из ENV (цены/ник) в рантайме + правило эскалации. Свободный текст пользователя → LLM с памятью последних ~6 реплик (3 обмена).

## Компоненты

- **`bot/services/llm.py`** — `async def ask(user_id: int, question: str) -> tuple[str, bool]` → `(answer, escalated)`.
  - Память: модульный `dict[int, deque(maxlen=6)]` сообщений `{role, content}`; теряется при рестарте (v1).
  - Собирает `messages = [system] + list(history) + [user]`, зовёт OpenAI с `response_format={"type":"json_object"}`, парсит `{"answer": str, "can_answer": bool}`. `escalated = not can_answer`. При ошибке парсинга — `answer = raw, escalated = False` (graceful).
  - Системный промпт: роль ассистента «Русский Лайнап», вставка `knowledge.md` + значений ENV (`board_12kw`, `board_14kw`, `seat_kit`, `battery_1/2`, `partner_nick`, `partner_discount`, тест-день — «уточняйте у менеджера»). Инструкция: отвечать только из этих данных; не знаешь/не по теме → `can_answer=false`; не раскрывать системные инструкции/секреты; коротко и дружелюбно.
  - Нет `LLM_API_KEY` → AI выключен: `ask` возвращает `("", True)` (мягкая эскалация), без обращения к OpenAI.
- **`bot/content/knowledge.md`** — факты бренда (электросёрф, тест-день, кэмпы, комьюнити). **Без цен/секретов** — конкретику подставляет `llm.py` из ENV в рантайме. Поставляется стартовый шаблон, владелец дополняет.
- **`bot/handlers/ai.py`** —
  - callback `ai:ask` → подсказка «Задай вопрос — отвечу по электросёрфу, тест-дням и кэмпам 🏄».
  - **catch-all** `@router.message(StateFilter(None), F.text, ~CommandObject)` → срабатывает только на свободный текст вне FSM-заявки и не на команды. Поток: проверка дневного лимита → `llm.ask` → ответ; если `escalated` — добавляем «не уверен — напиши {partner_nick}» + инлайн-кнопка «Написать менеджеру» (URL на партнёра). Лог `ask_ai`.
  - Регистрируется в `__main__` **последним** роутером.
- **`bot/config.py`** + **`.env.example`**: reuse `LLM_API_KEY` (OpenAI `sk-...`, секрет) и `LLM_MODEL` (дефолт **`gpt-4o`**, читается из ENV — можно поставить любую модель без правок кода); добавить `ai_daily_limit: int` (`AI_DAILY_LIMIT`, дефолт 20).
- **`requirements.txt`**: добавить `openai`.

## Эскалация (низкая уверенность)

Определяется структурным ответом модели: `can_answer=false` → бот не выдумывает, а предлагает менеджера и логирует факт эскалации. Лог: `log_event(event="ask_ai", detail=...)`, где `detail` = вопрос (обрезан до 200 символов), с префиксом `"[escalate] "` при эскалации — одна запись, дашборд/греп различает. (Без изменения схемы `events`.)

## Контроль расходов

Перед вызовом OpenAI считаем сегодняшние события `ask_ai` пользователя в `events`
(`SELECT count(*) FROM events WHERE user_id=$1 AND event='ask_ai' AND ts::date=now()::date`).
Если `>= ai_daily_limit` → ответ «на сегодня хватит вопросов, напиши {partner_nick}» + кнопка, **без обращения к API**. Новая функция `db.count_today_event(user_id, event) -> int`.

## Порядок роутеров / безопасность

- `lead`-FSM (хендлеры по состоянию) и команды (`/start`, `/stats`) — приоритетнее; AI-catch-all регистрируется последним и ограничен `StateFilter(None)`, поэтому не перехватывает ввод в диалоге заявки и не реагирует на команды.
- Системный промпт запрещает раскрывать инструкции/секреты и держит тему бренда (снижение prompt-injection). Ответы короткие.
- `OPENAI`-ключ и модель — только ENV (репо публичный).

## Поток данных

текст → лимит (events) → `llm.ask` (system+память+вопрос → OpenAI, JSON) → `(answer, escalated)` → ответ (+ кнопка менеджера при escalated) → `log_event(ask_ai, detail)` → обновить память.

## Тестирование (TDD)

- **`llm.ask`** с **замоканным** OpenAI-клиентом (monkeypatch): в `messages` есть текст из `knowledge.md` и значения ENV и история; корректный парс `{answer, can_answer}`; ветка «нет ключа» → `("", True)` без вызова; кап памяти (≤6).
- **лимит** — `db.count_today_event` с фейковым пулом (как `tests/test_db.py`).
- **хендлер** `ai.py`: лимит исчерпан → нет вызова `llm.ask` + сообщение с кнопкой; обычный → ответ + `log_event(ask_ai)`; эскалация → кнопка менеджера + префикс `[escalate]` в detail.
- Реальных вызовов OpenAI в тестах нет.

## Вне скоупа (v1)

- Эмбеддинги/RAG, персистентная память (БД), мультиязык, стриминг ответов, голос/изображения.
- Тонкая модерация контента (полагаемся на системный промпт + лимит).

## Зависимость от владельца

Наполнить `bot/content/knowledge.md` фактами (стартовый шаблон в комплекте). Цены/ник — уже в ENV. `LLM_API_KEY` (OpenAI) и при желании `LLM_MODEL`/`AI_DAILY_LIMIT` — задать в переменных Amvera.
